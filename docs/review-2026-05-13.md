# FlixMonkey — Deep Project Review

> Reviewed: 2026-05-13 | Version: 0.10.0

---

## Executive Summary

FlixMonkey is a well-structured, actively maintained multi-target extension with a clean Platform Adapter pattern and solid test scaffolding. The codebase is notably readable and consistent. The most pressing issues are: a **dual config system** that is half-migrated and leaking global mutable state, **cache reads that bypass `ApiClientManager`** causing double-reads in `app.js`, and **CI job duplication** that creates unnecessary overhead. Test coverage has gaps in critical path scenarios (error paths, rate-limit cross-tab sync, DOM mutation observer).

## 1. Build System

### 1.1 Findings

| # | Severity | Location | Finding |
|---|----------|----------|---------|
| B1 | **Major** | `rollup.config.js` | No source maps emitted for any target. Debugging production issues requires manual decompilation. |
| B2 | **Major** | `rollup.config.js` | Both Firefox and Chrome extension targets bundle `content.js` and `options.js` independently using `sharedPlugins()`. Each call re-instantiates `resolve()` + `commonjs()` plugins — fine functionally, but means no shared chunk between content/options bundles. Options code that imports the same modules will duplicate them. |
| B3 | **Minor** | `rollup.config.js:11` | `USERSCRIPT_BANNER` hardcodes `@namespace` to `https://github.com/fran/FlixMonkey`. If the repo URL ever changes, this and only this location needs updating — low risk but worth externalizing. |
| B4 | **Minor** | `rollup.config.js` | The `_target` property on config objects is a private convention that gets stripped before export. It works but is non-obvious. A comment or named-filter approach would be clearer. |
| B5 | **Minor** | `scripts/package.js` | `zipDirectory` uses `archive.pipe(output)` but doesn't return a Promise or handle errors beyond `output.on('close', ...)`. If archiving fails silently, `npm run build` still exits 0. |
| B6 | **Minor** | `package.json` | `dotenv` is listed as a **production dependency**, but it's only used in test/integration environment setup. Should be `devDependencies`. |
| B7 | **Minor** | `package.json` | No `engines` field. The project targets Node 22 (pinned in CI) but nothing prevents running builds under older Node versions. |

### 1.2 Proposed Improvements

```js
// rollup.config.js — add sourcemaps for non-userscript targets
output: {
  file: 'dist/firefox/content.js',
  format: 'iife',
  sourcemap: true,   // ← add
}
```

```js
// scripts/package.js — promisify and propagate errors
function zipDirectory(sourceDir, outPath) {
  return new Promise((resolve, reject) => {
    const output = fs.createWriteStream(outPath);
    const archive = archiver('zip', { zlib: { level: 9 } });
    output.on('close', () => { console.error(`Created ${outPath}`); resolve(); });
    archive.on('error', reject);
    archive.pipe(output);
    archive.directory(sourceDir, false);
    archive.finalize();
  });
}
// and await both calls with process.exit(1) on failure
```

```json
  "dotenv": "^17.4.2"   // ← move from dependencies
}
```

## 2. CI Pipeline

### 2.1 Findings

| # | Severity | Location | Finding |
|---|----------|----------|---------|
| C1 | **Major** | `ci.yml` | Every job independently runs `npm ci` — 5 separate full installs per push. A single install job that caches `node_modules` and uploads it as an artifact would save 4×install time (~30-60 s each). The current `needs: install` dependency is present but the `install` job itself doesn't upload artifacts, so all other jobs re-run `npm ci` redundantly. |
| C2 | **Major** | `ci.yml` | No **coverage reporting** step. `test:coverage` is defined in `package.json` but never called in CI. Coverage regressions are invisible to reviewers. |
| C3 | **Major** | `ci.yml` | `npm run audit` runs without `--audit-level` set, so it exits non-zero only for `critical` vulnerabilities by default. A high-severity vuln won't fail the build. Should add `--audit-level=high`. |
| C4 | **Minor** | `ci.yml` | `lint-and-audit` doesn't depend on `install`, so it could theoretically start before it completes (it re-installs anyway, so functionally fine, but architecturally inconsistent). |
| C5 | **Minor** | `release-please.yml` | Release Please workflow has no Node version pinned. If the runner's default Node changes, manifest parsing could break silently. |
| C6 | **Minor** | Both workflows | No `timeout-minutes` on any job. A hung test can block the queue indefinitely. |

### 2.2 Proposed Improvements

```yaml
# ci.yml — improved structure using upload-artifact for node_modules
install:
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-node@v4
      with:
        node-version: '22'
        cache: 'npm'
    - run: npm ci
    - uses: actions/upload-artifact@v4
      with:
        name: node-modules
        path: node_modules
        retention-days: 1

# All other jobs: replace "npm ci" with:
    - uses: actions/download-artifact@v4
      with:
        name: node-modules
        path: node_modules
```

```yaml
# Add coverage reporting job
test-coverage:
  runs-on: ubuntu-latest
  needs: install
  timeout-minutes: 10
  steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-node@v4
      with:
        node-version: '22'
        cache: 'npm'
    - run: npm ci
    - run: npm run test:coverage
    - uses: actions/upload-artifact@v4
      with:
        name: coverage
        path: coverage/
```

## 3. Source Code

### 3.1 Architecture

The three-layer architecture (core / platform / targets) is clean and well-reasoned. Private class fields (`#field`) are used consistently. The Platform Adapter pattern is correctly implemented — `httpFetch`, `storageGet`, `storageSet` are the only cross-cutting concerns.

### 3.2 Findings — Critical

| # | Severity | Location | Finding |
|---|----------|----------|---------|
| S1 | **Critical** | `src/core/app.js:51-53` | **Double cache read**: `#decorateContainer` first calls `this.#cache.read()` directly, then calls `this.#api.getData()` which itself calls `this.#cache.read()` again. The cache is read at least twice per title per request that misses. This is a latency and storage access waste. The `ApiClientManager.getData` already handles caching; `app.js` should not read the cache directly. |
| S2 | **Critical** | `src/core/app.js:73-74` | **Global state mutation**: `history._fmPatched` is a property set directly on the browser `history` object. This leaks into page global state, could conflict with other extensions/scripts, and bypasses class encapsulation. Should use a module-level boolean. |
| S3 | **Critical** | `src/core/app.js:115` | **Debug global leaked**: `window.fmApi = api;` is hardcoded in production code with no feature flag or dev-mode guard. This exposes the internal `ApiClientManager` to any page script running in the same context. |
| S4 | **Critical** | `src/core/config.js` | **Dual config system**: Both `CONFIG` (global singleton, `config.js`) and `ConfigManager` (class, `config-manager.js`) exist and solve the same problem. `app.js` instantiates `ConfigManager` and also calls `initConfig()` to wire the global `CONFIG`. Downstream modules receive `ConfigManager` instances while the old `CONFIG` object is still exported but only referenced in `config.test.js`. This is a half-completed migration — the `CONFIG` singleton should be removed and all callers unified under `ConfigManager`. |

### 3.3 Findings — Major

| # | Severity | Location | Finding |
|---|----------|----------|---------|
| S5 | **Major** | `src/core/api-manager.js:46-63` | **`isBetterThan` semantics are too weak**: the method only checks `!!this.rating && !other?.rating`. The `break` on first match means clients after the first one that returns a valid rating are never tried, even if a later client would provide RT/MC scores the first didn't have. Consider collecting all results and merging scores. |
| S6 | **Major** | `src/core/cache.js` | **Cache eviction on every write**: `write()` iterates all keys and deletes expired ones on every write operation. For large caches this is O(n) per write. Should be done lazily (periodic cleanup) or on read misses, not on every write. |
| S7 | **Major** | `src/core/request-queue.js:43-53` | **Sort on every dequeue iteration**: `this.#queue.sort(...)` is called inside the `while` loop before every item is processed. A priority queue (min-heap) would be O(log n) per insert vs O(n log n) sort per dequeue. At current volumes this is negligible, but is architecturally wrong. |
| S8 | **Major** | `src/core/api-clients.js:127` | **`logger.warn` for normal info messages**: debug-level messages like "Searching XMDB for title: …" are logged at `warn` level. Logger has an `info` method — these should use it. Only genuine warnings/errors should use `warn`/`error`. This causes noise in the browser console. |
| S9 | **Major** | `src/core/surfaces.js` | **No zero-surface telemetry**: if Netflix changes their DOM, `querySelectorAll` silently returns empty. The `try/catch` only catches selector syntax errors, not mismatches. Consider logging a `warn` when `discover()` returns empty on a known Netflix URL. |

### 3.4 Findings — Minor

| # | Severity | Location | Finding |
|---|----------|----------|---------|
| S10 | **Minor** | `src/core/overlay.js:28` | `injectStyles()` has no guard against being called twice. Repeated calls (e.g. on SPA navigation) would inject duplicate `<style>` tags. Add an id-based guard: `if (document.querySelector('#fm-styles')) return;`. |
| S11 | **Minor** | `src/core/title.js:56` | `isBetterThan` is an instance method but reads purely from `this` and `other` — could be a static comparator for clarity and testability. |
| S12 | **Minor** | `src/platform/webextension.js` | `WebExtensionAdapter` doesn't implement `registerMenuCommand`. The base class throws if called. Extensions don't use menu commands, so this should be a no-op override rather than a runtime-throw risk. |
| S13 | **Minor** | `src/core/constants.js` | `USER_AGENTS` is defined in `constants.js` but is only used by `UserscriptAdapter`. It is a platform-specific concern that bleeds into core. Should live in `src/platform/userscript.js`. |
| S14 | **Minor** | `src/core/api-manager.js:constructor` | Client instantiation from string config is done inline in the constructor with an anonymous `clientMap` object. Extracting to a named factory function (`createClientsFromConfig`) would make the constructor cleaner and the factory independently testable. |
| S15 | **Minor** | `src/core/disabled-clients.js` | `resetAll()` only resets the three known `ApiSource` values. If a new source is added, it's an implicit coupling that could be missed during refactoring. |
