# Automated Store Publishing: Design

**Date:** 2026-06-20
**Status:** Approved, pending implementation
**Branch:** `ci/publish-to-stores`

## Goal

Automatically publish each FlixMonkey release to the **Chrome Web Store** and
**Firefox Add-ons (AMO)**, extending the existing release pipeline. Both store
listings already exist, so this automates **version updates**, not first-time
submissions. Publishing is **fully automatic**: CI uploads the new version and
submits it for review with no manual click. Reviews are then handled by each
store on its own schedule.

## Scope

In scope:

- A new `publish-stores.yml` GitHub Actions workflow that publishes a release to
  both stores.
- A small change to `release-please.yml` to emit a `release-info` artifact when a
  release is cut.
- New npm scripts and pinned dev dependencies for the two store CLIs.
- An auto-generated source archive submitted to AMO alongside the signed build.
- Documentation for obtaining and storing the required credentials.

Out of scope:

- First-time store submissions (already done manually).
- Userscript distribution (already attached to the GitHub Release; not a store).
- Auto-rollback (stores do not support it; fix forward with a new patch release).

## Trigger Mechanism

GitHub does not trigger workflows from events generated by the built-in
`GITHUB_TOKEN`. release-please creates the GitHub Release using `GITHUB_TOKEN`,
so an `on: release: [published]` trigger would never fire. To avoid introducing
a long-lived PAT or GitHub App token (keeping the repository's minimal-secret,
pinned-SHA posture), publishing is triggered with a `workflow_run` handshake:

1. `release-please.yml`: when `steps.release.outputs.release_created` is true,
   after uploading assets, write the tag name to a file and upload it as an
   artifact named `release-info`.
2. `publish-stores.yml`: runs `on: workflow_run` for the `Release Please`
   workflow, `types: [completed]`. It gates on
   `github.event.workflow_run.conclusion == 'success'`, then attempts to
   download the `release-info` artifact.
    - **Artifact absent:** no release was cut this run (only a release-PR update).
      The workflow exits 0 and skips publishing.
    - **Artifact present:** read the tag and proceed.

Benefits:

- No extra secrets beyond the store credentials.
- The publish run is independently re-runnable from the Actions UI.
- A store failure never affects the release-please job.

A secondary `workflow_dispatch` trigger with a `tag` input is also included so a
specific tag can be published manually without re-running release-please.

## Workflow Structure (`publish-stores.yml`)

Permissions: `contents: read` and `actions: read` (the latter is required to
download an artifact produced by a different workflow run).

```
job: resolve-release        (runs first)
  - gate on workflow_run.conclusion == 'success' (skip for workflow_dispatch)
  - download the `release-info` artifact
  - if absent -> output skip=true
  - outputs: tag, skip

job: publish-chrome         (needs resolve-release, if skip != true)
job: publish-firefox        (needs resolve-release, if skip != true)
```

`publish-chrome` and `publish-firefox` run in **parallel**, so a Chrome
rejection does not block Firefox and each is re-runnable on its own.

Both publish jobs share a prelude:

1. `actions/checkout` at the release **tag** (from `resolve-release` output, or
   the `workflow_dispatch` input).
2. `actions/setup-node` with Node 24.
3. `npm ci --ignore-scripts`.
4. `npm rebuild sharp` (native dep needed for icon resizing at build time).
5. Build the relevant target.

Building from the tag guarantees the version injected into the manifest matches
the release, and (for Firefox) that the submitted source archive matches exactly
what is published. All third-party actions are pinned to commit SHAs, matching
the existing workflows.

## Chrome Job

- `npm run build:chrome` produces `dist/chrome/` and
  `dist/FlixMonkey-v<version>-chrome.zip`.
- `npm run publish:chrome` wraps `chrome-webstore-upload-cli`:
    - `chrome-webstore-upload-cli upload --source dist/FlixMonkey-v<version>-chrome.zip --auto-publish`
    - Credentials passed as environment variables from the `CHROME_*` secrets.
- `chrome-webstore-upload-cli` is pinned as a devDependency in `package.json`.
- Wrapping the CLI in an npm script keeps it runnable and testable locally.

## Firefox Job

- `npm run build:firefox` produces `dist/firefox/`.
- Build a **source archive** of the repository (via `git archive` or the existing
  `archiver` dependency) excluding `node_modules`, `dist`, `coverage`, and `.git`,
  plus a short reviewer note describing the reproducible build:
  `npm ci && npm run build:firefox`. AMO requires source for listed add-ons that
  ship a build step.
- `npm run publish:firefox` submits the build to AMO's **listed** channel via
  Mozilla `web-ext sign` (pinned devDependency) using `AMO_JWT_ISSUER` and
  `AMO_JWT_SECRET`, attaching the source archive.

**Verification item (resolve during planning):** confirm the exact mechanism the
current `web-ext` version exposes for attaching the source archive
(`web-ext sign` source option vs. a follow-up AMO API `PATCH` on the created
version). The AMO submission API supports source upload; the implementation must
use whichever path the pinned CLI version supports.

**Add-on ID:** AMO requires a stable add-on ID for version updates. Verify
`src/targets/firefox/manifest.json` sets
`browser_specific_settings.gecko.id`; if missing, it must be added (this is a
prerequisite for automated updates).

## Secrets and Credential Setup

Documented in the spec/docs so a maintainer can configure them.

### Chrome Web Store

Repository secrets: `CHROME_EXTENSION_ID`, `CHROME_CLIENT_ID`,
`CHROME_CLIENT_SECRET`, `CHROME_REFRESH_TOKEN`.

1. Create (or reuse) a Google Cloud project and enable the **Chrome Web Store
   API**.
2. Create an OAuth2 **Desktop app** client; record the client ID and secret.
3. Run the one-time consent flow to mint a **refresh token** (documented commands
   to be included in the implementation docs).
4. `CHROME_EXTENSION_ID` is the existing item's ID from the Web Store dashboard.

### Firefox AMO

Repository secrets: `AMO_JWT_ISSUER`, `AMO_JWT_SECRET`.

1. Generate API credentials at `addons.mozilla.org` -> Manage API Keys.
2. The add-on ID/GUID comes from the Firefox manifest's
   `browser_specific_settings.gecko.id`.

## Failure Handling and Idempotency

- Each store job is independent and re-runnable; failures surface per-job.
- **Re-publish guard:** store APIs reject re-uploading an existing version. Where
  detectable, the scripts treat "version already exists" as a non-fatal skip;
  otherwise the job fails loudly and only the incomplete job is re-run.
- **No auto-rollback:** stores do not support it. A bad release is fixed by
  cutting a new patch version.

## Testing and Validation

- CI cannot end-to-end test real store uploads.
- Validation consists of:
    - Workflow YAML lints and parses.
    - The new npm scripts run locally in a `--dry-run` / no-publish mode where the
      CLIs support it.
    - A first manual `workflow_dispatch` run validates against the live stores once
      secrets are configured.
