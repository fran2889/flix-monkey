# Design Spec: Per-Item Cache Storage

## Overview

FlixMonkey currently stores all title ratings in a single JSON blob under the `fm_cache` storage key. As the cache grows (up to 10,000+ items), this approach becomes a performance bottleneck due to frequent serialization/deserialization and large write operations on every cache hit. This design moves to a per-item storage strategy.

## Goals

- Improve performance by moving from O(N) to O(1) read/write operations.
- Avoid browser storage limits for single keys.
- Reduce CPU and I/O overhead during browsing.

## Architecture

### 1. Storage Key Strategy

Cache entries will be stored using a prefixed key format:
`fmc:<title_key>`

Example: `fmc:the_matrix_1999`

The `title_key` will continue to be generated by the existing `#getCacheKey(displayTitle, domYear)` logic (lowercase, spaces to underscores).

### 2. Platform Adapter Updates

The `PlatformAdapter` interface and its implementations (`UserscriptAdapter`, `WebExtensionAdapter`) will be expanded to support listing and deleting individual keys.

New methods:

- `storageDelete(key)`: Deletes a specific key from storage.
- `storageGetKeys(prefix)`: Returns an array of all storage keys starting with the specified prefix.

#### Implementation Details:

- **WebExtension:**
    - `storageDelete`: Uses `browser.storage.local.remove(key)`.
    - `storageGetKeys`: Uses `browser.storage.local.get(null)` to retrieve all keys and filters them. While `get(null)` is heavy, it will only be called during `clear()` operations, not during normal browsing.
- **Userscript:**
    - `storageDelete`: Uses `GM_deleteValue(key)`.
    - `storageGetKeys`: Uses `GM_listValues()` and filters by prefix.

### 3. CacheManager Refactoring

- `read(displayTitle, domYear)`: Performs a single `storageGet` for the specific title key.
- `write(displayTitle, domYear, titleObj)`: Performs a single `storageSet` for the specific title key. The "automatic purge" of all expired entries on every write will be removed.
- `clear()`: Retrieves all keys starting with `fmc:` and deletes them individually or via a bulk remove API if supported.

### 4. Migration Strategy

No migration will be performed. Existing data in the `fm_cache` key will be ignored by the new implementation. Users who wish to clear the old data can do so by manually clearing their browser storage or using a one-time console command if needed.

## Data Flow

1. `App` requests rating for "The Matrix".
2. `CacheManager` generates key `fmc:the_matrix_1999`.
3. `CacheManager` calls `adapter.storageGet('fmc:the_matrix_1999')`.
4. If found and not expired, returns data.
5. If not found, `App` fetches from API.
6. `CacheManager` calls `adapter.storageSet('fmc:the_matrix_1999', data)`.

## Error Handling

- If a cache entry is corrupted (failed `JSON.parse`), it will be treated as a cache miss.
- Expired entries are detected during `read()` and ignored.

## Testing Strategy

- Update `tests/unit/core/cache.test.js` to mock the new adapter methods.
- Verify that `write()` no longer triggers a full cache read.
- Verify that `clear()` correctly identifies and removes only `fmc:` prefixed keys.
- Add a test case for "stale" entries (expired items that haven't been overwritten).
